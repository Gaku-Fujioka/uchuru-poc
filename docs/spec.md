# uchuru デモアプリ仕様書

更新日: 2026年6月25日  
対象: 観光クロスオーバーコンテスト2026 / デモアプリ制作 / スライド作成前整理

## 1. 目的

この仕様書は、現行リポジトリの React/Vite 製モックアプリを、観光クロスオーバーコンテスト2026向けの `uchuru` デモアプリへ作り替えるための実装指針をまとめる。

`uchuru` は、宿泊予約サービスではない。観光で大阪を好きになった人を、1週間の生活体験を通じて、地域と関わり続ける候補者に変える実証システムとして見せる。

中心コピー:

> 「住んでみる」を、観光の次の一歩に。

最重要メッセージ:

> uchuruは、宿泊サービスではありません。  
> 観光で大阪を好きになった人を、1週間の生活体験を通じて、地域と関わり続ける候補者に変える実証システムです。

## 2. 今回の再構成ポイント

前回仕様から、以下の方向へ絞り込む。

- ターゲットを `国内外の若者・ノマドワーカー` に絞る。
- 滞在期間を `1週間〜1ヶ月` に絞る。
- 実証エリアを `なんば・天王寺周辺` に絞る。
- 地域情報の単なる案内ではなく、`大阪生活実感ミッション` として体験設計する。
- 成功指標を、予約数ではなく `生活ミッション実行率` と `関係人口候補化率` で見せる。
- 決済、契約、予約確定、鍵管理、清掃管理、AIチャット、スマートロックは今回のMVPから外す。
- 管理者向け運用画面は作らない。ただし、発表デモの最後にKPIを見せるための `PoCダッシュボード` は作る。

補足: 調整メモでは「管理者ダッシュボード」と書かれているが、以前の方針である「管理者画面は作らない」と矛盾しないよう、実運用の管理画面ではなく、発表用の検証結果可視化画面として扱う。

## 3. アプリの役割

デモアプリの目的:

> uchuruは、ただ宿を探すアプリではなく、「1週間だけ大阪に住んでみる」ために、滞在先・生活ミッション・地域接点・関係人口候補化の測定までできるアプリであることを見せる。

デモで伝えること:

1. 大阪に1週間だけ住んでみる入口がある。
2. なんば・天王寺周辺の滞在先を選べる。
3. 物件情報だけでなく、働く・暮らす・関わるミッションがある。
4. 滞在後アンケートで、再訪・長期滞在・地域情報受信希望を測る。
5. PoCダッシュボードで、関係人口候補化率を可視化できる。

## 4. 課題設定

観光で大阪を好きになっても、その次の選択肢は少ない。

現状の選択肢:

1. 数日だけ泊まって帰る。
2. いきなり賃貸契約や移住を検討する。

本来必要な中間ステップ:

- 1週間だけ住んでみる。
- 生活や仕事のしやすさを試す。
- 地域と関わってみる。
- もっと長く滞在したいかを判断する。

uchuruが解く課題:

> 観光と移住の間にある「試し住み」の空白。

## 5. 初期ターゲット

確定ターゲット:

> 大阪に1週間〜1ヶ月住んでみたい国内外の若者・ノマドワーカー。

プレゼン向け表現:

> 大阪での生活・仕事を試したい若手社会人・学生・外国人ノマド。

想定ユーザー:

- 国内大学生
- 若手社会人
- フリーランス/ノマドワーカー
- 外国人ノマド
- 留学生や国際交流コミュニティの参加者

広げすぎないため、初期実証では「移住検討者」「観光客」「留学生」などを全部並べるのではなく、`1週間大阪で暮らしてみたい国内外の若者・ノマド` に寄せる。

## 6. 初期実証エリア

確定エリア:

- なんば
- 天王寺
- その周辺エリア

選定理由:

- 観光地として訪れやすい。
- 交通アクセスが良い。
- 生活に必要な店が多い。
- 宿泊施設や簡易宿所が多い。
- 若者、外国人、ノマドとの相性が良い。
- 観光と生活の両方を試しやすい。

プレゼン用表現:

> 初期実証は、観光・生活・交通の接続が強いなんば・天王寺周辺で行います。

| 鍵管理 | 初期PoCでは事業者側が担うため |
| 清掃管理 | 初期PoCでは事業者側が担うため |
| AIチャット | 今回の本質ではないため |
| スマートロック | 初期PoCでは不要なため |
| 本格ログイン | デモでは重く、価値説明に直結しないため |
| 複雑な検索フィルター | 宿泊予約アプリに見えやすいため |
| 実運用の管理者画面 | 今回はフロントデモとPoC可視化に絞るため |

### 10.5 事前登録フォーム画面

目的:

- ただの興味ではなく、予算・時期・有料滞在意思を含む質の高いリードを取る。

入力項目:

- 年齢
- 属性
- 学生 / 社会人 / 外国人 / ノマド
- 大阪に住んでみたい理由
- 希望滞在期間
- 希望時期
- 予算上限
- 実際に有料で滞在する意思
- 使いたいミッション
- 連絡先

重要な設計:

- `興味あります` だけで終わらせない。
- 予算、時期、有料滞在意思を必須に近い形で聞く。
- 登録後に `PoCで検証したい項目` が分かる完了画面を出す。

完了文言例:

> 事前登録ありがとうございます。  
> uchuruは、実際に有料で1週間大阪に住んでみたい人がいるかを検証しています。  
> 実証開始時に、なんば・天王寺周辺の滞在候補と生活ミッションをご案内します。

### 10.6 滞在後アンケート画面

目的:

- 関係人口候補化を測る。
- 予約数・宿泊数ではなく、地域との継続接点を評価する。

質問:

| 質問 | 選択肢 |
| --- | --- |
| 大阪にまた来たいですか？ | はい / 迷う / いいえ |
| 次はもっと長く滞在したいですか？ | はい / 迷う / いいえ |
| 大阪の地域情報を今後も受け取りたいですか？ | はい / いいえ |
| 地域イベントや地元店にまた関わりたいですか？ | はい / いいえ |
| 将来的に大阪を二地域居住・移住先として考えられますか？ | はい / 迷う / いいえ |

関係人口候補の定義:

滞在後に以下のいずれかを希望した人を `関係人口候補` とする。

- 再訪したい
- 次はより長く滞在したい
- 地域情報を継続して受け取りたい
- 地域イベントに参加したい
- 生活情報オリエンテーションを受けたい

結果表示例:

> あなたは「大阪の関係人口候補」です。  
> 次回の長期滞在情報や地域イベント情報を受け取れます。

## 13. KPI

3ヶ月PoCの主要KPI:

| 項目 | 目標 |
| --- | ---: |
| 営業・打診先 | 30件 |
| 提携事業者 | 3〜5社 |
| 掲載物件 | 10〜20室 |
| 事前登録 | 100人 |
| 問い合わせ | 20〜30件 |
| 実滞在 | 3〜5件 |
| 大阪生活実感セット利用率 | 50%以上 |
| 関係人口候補化率 | 60%以上 |

追加検討KPI:

| 項目 | 目標 | 理由 |
| --- | ---: | --- |
| 有料滞在意思あり率 | 事前登録者の30%以上 | 無料興味層を除外するため |
| 予算一致率 | 事前登録者の30%以上 | 価格とターゲットのズレを見るため |
| 地域接点実行数 | 滞在者1人あたり3件以上 | マップ閲覧だけで終わらせないため |
| 滞在後フォロー希望率 | 50%以上 | 継続接点を見るため |
| 事業者継続意向 | 提携先の半数以上 | 事業者側の価値を見るため |

3分プレゼン用:

> 3ヶ月で、営業30件、提携3〜5社、掲載10〜20室、事前登録100人、実滞在3〜5件を目指します。  
> さらに、滞在者の50%以上が生活ミッションを実行し、60%以上が再訪・長期滞在・地域情報の継続受信を希望するかを検証します。

## 16. 収益モデル

### Phase 1: 送客手数料

既存事業者に中期滞在者を送客し、成約額の5〜10%を手数料として受け取る。

特徴:

- 物件を借り上げない。
- 在庫リスクを持たない。
- 学生チームでも始めやすい。
- 初期PoCと相性が良い。

### Phase 2: 中期滞在特化SaaS

成長後は、事業者向けに中期滞在特化の管理機能を提供する。

機能例:

- 多言語FAQ
- 施設利用・周辺生活情報の多言語案内
- 滞在前オリエンテーション
- 地域接点利用ログ
- 滞在後アンケート
- 再訪・長期滞在見込み客の管理
- 管理ダッシュボード

### Phase 3: 地域接点オプション

将来的には以下のオプション収益を検討する。

- コワーキング利用
- 地元店クーポン
- 生活情報オリエンテーション
- 地域イベント参加
- 長期滞在相談

### Phase 4: 限定的マスターリース

需要データが取れた物件に限って、より深い運営関与や限定的マスターリースを検討する。

ただし、初期には行わない。空室リスクが大きいため、最初から物件を借り上げない。

## 17. 法務・オペレーション設計

基本方針:

- 最終形では、物件探しから契約・決済・入退去・地域接点までアプリで完結する体験を目指す。
- 初期PoCでは法務リスクを抑えるため、契約・決済はuchuru側で持たない。

初期PoCでuchuruが担うこと:

- 物件情報掲載
- 各事業者の予約・問い合わせページへの送客導線
- 滞在UX
- 大阪生活実感セット
- ユーザー獲得
- アンケート分析

提携事業者が担うこと:

- 契約
- 決済
- 鍵管理
- 清掃
- リネン
- キャンセル・返金対応
- 設備不良対応
- 最終トラブル対応

30日未満:

- 旅館業許可、特区民泊認定、住宅宿泊事業届出などを持つ施設を対象にする。

30日以上:

- 定期建物賃貸借などが関係する可能性があるため、事業者とユーザーの直接契約を前提にする。
- uchuruは契約の仲介・代理を行わない。

危ない表現と安全な表現:

| 危ない表現 | 安全な表現 |
| --- | --- |
| 問い合わせ導線 | 各事業者の予約・問い合わせページへの送客導線 |
| 生活相談 | 生活情報オリエンテーション |
| 契約を支援する | 必要情報を整理する |
| 決済をアプリで完結 | 初期は決済・契約は事業者とユーザー間で行う |
| 契約自動切替 | 30日未満/以上の滞在形態を整理する設計 |
| 外国人の契約を代行 | 多言語で必要情報を案内し、必要に応じて専門家につなぐ |

質疑での安全回答:

> 初期PoCでは、uchuruは宿泊契約や賃貸契約の主体にはなりません。  
> 30日未満は旅館業許可、特区民泊認定、住宅宿泊事業届出などを持つ施設に限定し、契約・決済・鍵管理・清掃・最終トラブル対応は事業者側が行います。  
> uchuruは、各事業者への送客導線、滞在UX、生活情報オリエンテーション、アンケート分析に絞って検証します。

## 18. 競合との差別化

競合:

- Airbnb
- goodroom
- ADDress
- HafH
- マンスリーマンション / W&M
- Sakura House
- unito

注意点:

`1週間〜1ヶ月泊まれる` だけでは差別化にならない。Airbnbにも長期滞在があり、goodroomも暮らし文脈が強く、ADDressも多拠点・地域滞在の文脈を持っている。

差別化の3層:

1. 目的の違い  
   既存サービスは、泊まる場所・住む部屋・多拠点滞在を提供する。uchuruは、観光で関心を持った地域を `住む候補地` として試す。

2. 体験設計の違い  
   uchuruは物件だけでなく、働く・暮らす・関わるミッションを提供する。

3. 測定する指標の違い  
   既存サービスは、予約数・宿泊日数・稼働率・売上・レビューを見る。uchuruはそれに加えて、生活ミッション実行率、再訪意向、長期滞在意向、地域情報受信希望まで測る。

最終差別化コピー:

> 予約で終わるのが既存サービス。  
> 関係人口候補化まで測るのがuchuru。

## 20. 完成イメージ

MVP完成時、デモアプリは以下のストーリーを短時間で伝える。

1. 大阪に1週間だけ住んでみる。
2. なんば・天王寺の滞在先を選ぶ。
3. 物件だけでなく、働く・暮らす・関わるミッションが出る。
4. 事前登録で、予算・時期・有料滞在意思を取る。
5. ミッション実行で、Googleマップとは違う行動設計を見せる。
6. 滞在後アンケートで、再訪・長期滞在・地域情報受信希望を測る。
7. PoCダッシュボードで、関係人口候補化率を見せる。

この状態になれば、uchuruは単なる宿泊アプリではなく、観光関心から1週間滞在、生活ミッション、地域接点、関係人口候補化までを設計する実証システムとして伝わる。
